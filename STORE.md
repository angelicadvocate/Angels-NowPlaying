# Angels NowPlaying — Overlay Store: Implementation Notes

This document covers the design and implementation plan for the community overlay store.
It is intended to travel with the store repo when that repo is created.

The store is a separate site (GitHub Pages) embedded in the app as an iframe.
The app handles all installs; the store site is a read-only directory.

---------------------------------------------------------------------------------

## Hosting Model

- **GitHub Pages** off a dedicated catalog repo (separate from this app repo)
- Static files only — no server-side code
- Custom domain when ready (add a `CNAME` file + DNS record)
- GitHub Actions handle all "backend-like" work (validation, catalog generation) at deploy time
- Costs nothing to host; deploys automatically on every push to the target branch

---------------------------------------------------------------------------------

## Repo Structure

```
angels-nowplaying-store/        ← the catalog repo
  overlays/
    frame-my-overlay/
      meta.json                 ← one file per overlay submission
    frame-another-overlay/
      meta.json
  catalog.json                  ← AUTO-GENERATED by CI — do not edit manually
  index.html                    ← the store site UI
  style.css
  store.js
  README.md                     ← submission instructions for authors
```

`catalog.json` is always regenerated from all `meta.json` files on every push.
Authors only ever touch their own `meta.json`.

---------------------------------------------------------------------------------

## meta.json Schema

Each overlay submission is a single `meta.json` file. All fields are required unless marked optional.

```json
{
  "id": "frame-my-overlay",
  "name": "My Overlay",
  "version": "1.0.0",
  "author": "AuthorName",
  "description": "Short description shown in the store listing.",
  "licenseType": "MIT",
  "isFree": true,
  "isAnimated": false,
  "downloadUrl": "https://github.com/author/repo/releases/latest/download/frame-my-overlay.zip",
  "thumbnailUrl": "https://raw.githubusercontent.com/author/repo/main/preview.png",
  "tags": ["minimal", "dark"],
  "externalUrl": null
}
```

**Field notes:**
- `id` — must be unique across all entries. Same value as the `id` in the overlay's `manifest.json`. Validated by CI on every PR.
- `version` — should match the `version` in the overlay's `manifest.json`. Used by the app to compare against the installed version.
- `isFree` — `true` if the overlay is free to install, `false` if it requires payment or an external purchase step. Used for store filtering. When `false`, `externalUrl` is implicitly required. Do not include a price value — prices change and will go out of sync; if authors want to advertise a price they should put it in `description` or the `externalUrl` destination.
- `isAnimated` — `true` if the overlay contains CSS animations, video, or other motion elements. Used for store filtering. Useful for users on low-spec machines or those who prefer static overlays.
- `downloadUrl` — direct link to the overlay zip. Can be a GitHub release asset, any CDN, or any stable URL. The catalog does not host the file itself.
- `thumbnailUrl` — direct image URL for the store listing card. Authors host this themselves.
- `licenseType` — human-readable label only (e.g. "MIT", "CC BY 4.0", "Paid"). Not enforced.
- `externalUrl` — **optional**. When present and non-null, the app opens this URL in the system browser instead of triggering the in-app install flow. Use this for paid overlays or overlays hosted behind a paywall. The store listing should visually distinguish these entries (e.g. "Get Overlay →" button instead of "Install").
- `tags` — optional array. Used for filtering/search on the store site.

---------------------------------------------------------------------------------

## Paid / External Overlays

Authors who want to monetize their overlays can set `externalUrl` to their Gumroad page,
personal site, or any payment URL. The catalog simply lists them; the app opens the URL
in the system browser and never attempts a zip download. No special approval process needed —
just a visually distinct listing so users know they are leaving the app.

---------------------------------------------------------------------------------

## Submission Flow (PR-based)

1. Author forks the catalog repo
2. Creates `overlays/<their-overlay-id>/meta.json`
3. Opens a PR
4. GitHub Actions runs validation (see below)
5. Maintainer reviews and merges
6. CI regenerates `catalog.json` and deploys to GitHub Pages automatically

Authors need a GitHub account. If this becomes a friction point for less technical users,
a Google Form → manual meta.json addition is a viable fallback for early submissions.

---------------------------------------------------------------------------------

## GitHub Actions: Validation + Catalog Generation

Two workflows needed:

**1. PR Validation (`validate.yml`) — runs on every PR**
- Parse every `meta.json` in `overlays/`
- Check all required fields are present and non-empty
- Check `id` uniqueness across all entries (primary collision guard)
- Check `downloadUrl` is a valid URL format
- Optionally: HEAD request to `downloadUrl` to verify the zip is reachable
- Fail the PR check if any validation fails; post a descriptive error comment

**2. Deploy (`deploy.yml`) — runs on push to main**
- Regenerate `catalog.json` from all `meta.json` files
- Commit `catalog.json` back to the repo (or use an artifact)
- Deploy to GitHub Pages

## In-App Integration

The store is embedded in the app as an iframe on the store page.

**Install flow (free overlays):**
1. User clicks Install on a listing in the iframe
2. The store site posts a `postMessage` to the parent window: `{ type: 'install-overlay', downloadUrl: '...' }`
3. The app's store page JS receives the message and calls the Tauri `install_overlay` command with the URL
4. App downloads the zip to a temp path, then runs the existing install logic

**External/paid overlay flow:**
1. User clicks "Get Overlay" on an external listing
2. Store site posts: `{ type: 'open-external', url: '...' }`
3. App calls Tauri's `shell::open` to open the URL in the system browser

**Version comparison:**
- The app reads the installed overlay's `version` from its local `manifest.json`
- The catalog entry's `version` is available from `catalog.json`
- The store listing (or the editor header — see app TODO) can surface both so the user can tell if an update is available

## Licensing Model

The catalog repo is metadata-only. It does not host overlay files and asserts no license
over the overlays it lists. Each overlay lives in the author's own repo under whatever
license they choose. The `licenseType` field in `meta.json` is informational only.

This means:
- Authors retain full ownership and licensing control
- The catalog's own license covers only the site code and catalog infrastructure
- No legal ambiguity from a mono-repo that might imply a single license for all content

## Open Questions / Future Considerations

- **Update notifications**: once the auto-updater is wired into the app, consider also
  checking the catalog for overlay version bumps and surfacing an "update available" badge
  on the store listing or in the overlay list on the index page.
- **Categories/tags**: the `tags` field is there from day one, but the store UI filtering
  can be added later once there are enough overlays to make filtering useful.
- **Download counts / popularity**: GitHub doesn't expose download counts for raw file URLs.
  If this matters later, authors could use a service like `shields.io` or a GitHub release
  asset (which does have a download count via the GitHub API).
- **PR bot / automation**: a bot that auto-comments a preview card on the PR ("here's what
  your listing will look like") would improve the author submission experience significantly.
