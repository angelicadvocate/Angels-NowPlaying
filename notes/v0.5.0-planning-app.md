# Angels Now Playing v0.5.0 — Desktop Configuration Manager (Tauri App)

## Project Overview

The v0.5.0 release will extend Angels Now Playing from a collection of static OBS browser sources into a **desktop configuration manager** built with Tauri. Users will run this app locally to manage settings, customize themes, and generate/update configuration files for the 8 existing Now Playing widget frames (F1–F8).

The desktop app is the interface; the existing HTML/CSS/JS frames remain the delivery mechanism for OBS.

### Core Purpose

Provide a **lightweight, single-user desktop application** that:
- Centralizes configuration for all 8 widget templates
- Manages CSS customization, color schemes, and animation tuning
- Automatically writes updated configs back to disk (no "Save As" dialogs)
- Exports widgets as local HTML files ready for OBS consumption
- Eventually runs on Windows, macOS, and Linux without code changes

---

## Architecture & Technology Stack

### Frontend
- **HTML/CSS/JavaScript** — all UI logic lives in static web files
- Reuse the existing editor templates from `editor_assets/` as a foundation
- Single-page app (SPA) or multi-tab interface for managing all 8 frames
- Real-time preview of widget changes (future enhancement)

### Backend (Rust + Tauri)
- Minimal native layer focused on file I/O and OS integration
- Responsibilities:
  - Accept messages from frontend JS
  - Read and write configuration files (CSS, JSON)
  - Manage platform-specific paths (AppData, ~/.config, etc.)
  - Handle folder operations and exports
- **No significant business logic in Rust** — mostly file operations

### Why Tauri

Tauri provides:
- **Cross-platform support** for Windows, macOS, and Linux with a single codebase
- **Lightweight runtime** using the system's native WebView (no Electron bloat)
- **Simple JS ↔ Rust communication** for backend calls
- **Built-in file I/O** without triggering OS dialogs
- **Small installer size** and low memory footprint
- **Auto-update support** for future releases

## Data & File Storage

### Directory Structure
Configuration files are stored under the user's application data directory:

- **Windows:** `%APPDATA%/AngelsNowPlaying/`
- **macOS:** `~/Library/Application Support/AngelsNowPlaying/`
- **Linux:** `~/.config/AngelsNowPlaying/`

### Per-Widget Organization
Each of the 8 widget frames gets its own configuration folder:

```
AngelsNowPlaying/
  ├── Frame-F1/
  │   ├── config.json
  │   ├── styles.css
  │   └── widget-F1.html (exported)
  ├── Frame-F2/
  │   ├── config.json
  │   ├── styles.css
  │   └── widget-F2.html (exported)
  ├── ...
  └── Frame-F8/
      ├── config.json
      ├── styles.css
      └── widget-F8.html (exported)
```

### Configuration Files

**config.json** (per frame)
- Theme color palette
- Font family and sizes
- Animation speeds and timing
- Scroller behavior tuning (`--scroll-extra`, `--scroll-start-offset`)
- Content padding and alignment
- Any other CSS variable values

**styles.css** (per frame)
- Generated from config.json and template
- CSS custom properties (`--color-primary`, `--font-size`, etc.)
- Frame-specific overrides
- May reference a global base CSS or stay self-contained

### Global Settings
Optional app-level configuration:
- Default theme selection
- OBS export folder preference
- Auto-sync settings
- UI preferences (light/dark mode for the app itself)

## Frontend Implementation

### Entry Point & Navigation
- Single `index.html` that loads all UI components
- Navigation menu to switch between the 8 frames
- Each frame gets a dedicated section or view

### Editor UI Components
- **Frame Selector** — dropdown or tabs to pick which widget to edit
- **Live Preview Panel** (future) — shows how the widget will look
- **Color Picker** — for theme customization
- **Font Selector** — for typography adjustments
- **Scroller Tuning Panel** — sliders/inputs for `--scroll-extra`, `--scroll-start-offset`, padding
- **CSS Editor** — optional text area for custom CSS
- **Export / Apply Button** — writes config to disk

### Interaction Flow
1. User selects a frame (F1–F8)
2. App loads that frame's current `config.json`
3. UI populates with current settings
4. User adjusts settings in the interface
5. Changes trigger Tauri backend calls to write updated `config.json` and `styles.css`
6. (Optional) Preview updates in real-time
7. User clicks "Export for OBS" → generates `widget-F*.html` with embedded styles

---

## Backend Implementation (Tauri)

### Rust Responsibilities
1. **File Read** — load `config.json` for a given frame
2. **File Write** — save updated `config.json` and `styles.css`
3. **Path Resolution** — return the user's app data directory
4. **Export** — bundle frame HTML with updated CSS and output to user-selected location
5. **Folder Operations** — list frames, create new configs, delete old ones

### Tauri Commands (Rust → JS Interface)
Example commands exposed to JavaScript:

```rust
#[tauri::command]
async fn get_app_data_path() -> Result<String, String>

#[tauri::command]
async fn load_frame_config(frame_id: &str) -> Result<FrameConfig, String>

#[tauri::command]
async fn save_frame_config(frame_id: &str, config: FrameConfig) -> Result<(), String>

#[tauri::command]
async fn export_widget(frame_id: &str, export_path: &str) -> Result<(), String>

#[tauri::command]
async fn list_frames() -> Result<Vec<String>, String>

#[tauri::command]
async fn delete_frame_config(frame_id: &str) -> Result<(), String>
```

### Configuration Serialization
- Use `serde` + `serde_json` to parse/write `config.json`
- Define a Rust struct `FrameConfig` that mirrors the JSON schema
- CSS generation can be a simple string template or a library like `tera`

## Workflow: Setting Up & Running

### Development Setup
1. Create a new Tauri project: `cargo tauri init`
2. Copy existing widget HTML/CSS/JS into the Tauri `src-tauri/` and `src/` folders
3. Implement Rust backend commands for file I/O
4. Build the HTML/CSS/JS frontend UI
5. Connect frontend to backend via `tauri::invoke()`
6. Test locally with `cargo tauri dev`

### Build & Distribution
1. **Windows:** `cargo tauri build` → produces `.msi` and `.exe`
2. **macOS:** `cargo tauri build` → produces `.app` and `.dmg`
3. **Linux:** `cargo tauri build` → produces `.deb`, `.AppImage`, or `.rpm`

### User Installation & Runtime
1. User downloads installer for their OS
2. Runs installer (e.g., `AngelsNowPlaying-Setup.msi` on Windows)
3. App is installed to standard locations (Program Files, /opt, etc.)
4. User launches the app
5. App initializes config folders in their home directory if needed
6. User manages widget settings and exports files to OBS

## Updates & Distribution

### Update Strategy
**Version Control:** GitHub Releases  
The app uses GitHub as the single source of truth for releases. Each release is tagged, built, signed, and uploaded to GitHub Releases.

**Update Mechanism:** Tauri Updater  
Tauri's built-in updater handles:
- Checking for new versions
- Downloading updates from GitHub
- Verifying cryptographic signatures (automatic security)
- Installing updates with minimal user intervention
- Rollback on failure

### Implementation Steps

**Phase 1 (v0.5.0): Basic Update Checks**
1. Configure `tauri.conf.json` with updater enabled and GitHub Releases endpoint
2. Generate signing keys on first build (`secret.key` and `public.key`)
3. Add "Check for Updates" button in app UI
4. Users manually trigger checks; download and restart on new version
5. Display current version and latest available version in settings

**Phase 2 (v0.5.1+): Enhanced Update Experience**
1. Add "Check on app startup" option (optional, off by default)
2. Show update progress and download status in UI
3. Display release notes from GitHub before installing
4. Add changelog view in the app

**Phase 3 (v0.6.0+): Marketplace Integration**
1. When downloading custom overlays from the store, verify integrity
2. Update overlay version tracking in configs
3. Notify users of overlay updates separately from app updates
4. Automate overlay installation/removal from UI

### Release & Distribution Workflow
1. Developer pushes code to `main` branch
2. CI/CD builds artifacts for Windows/macOS/Linux
3. Tauri signs each artifact with `secret.key`
4. Upload artifacts and `latest.json` to GitHub Releases
5. Users' apps check GitHub API; download and install if newer version available
6. On next app launch, new version runs automatically

### GitHub Releases Configuration
Each release includes:
- Release notes (what's new, bug fixes, changes)
- Signed `.msi` (Windows)
- Signed `.dmg` (macOS)
- Signed `.AppImage` (Linux)
- `latest.json` manifest with version, signatures, and download URLs

### Security
- Private signing key (`secret.key`) stored securely (never committed to repo)
- Public key (`public.key`) bundled with app
- All updates verified before installation
- Failed updates don't break existing app installation

## Key Technical Considerations

### OBS Compatibility
- Exported HTML files must work as-is when pointed to as a local browser source in OBS
- Ensure CSS/JS paths are relative or self-contained (no external network calls except polling Song.json)
- Test exported files in OBS after each generation

### Data Persistence
- All configs are JSON files on disk; no database needed
- Version the config schema to support future changes
- Provide migration logic if schema changes in future releases

### Cross-Platform Testing
- Windows: Use `.msi` installer; verify file paths and shortcuts
- macOS: Ensure app signing/notarization requirements (future)
- Linux: Test `.deb` and `.AppImage` formats

### Security & Sandboxing
- Tauri sandboxes the app by default; explicit permissions needed for file I/O
- Restrict file access to app data directory and OBS export locations
- Validate all user input before writing to disk

### Performance
- Keep the frontend responsive during file operations (use async/await)
- Lazy-load frame configs if managing many frames
- Cache loaded configs to avoid repeated disk reads
